Git分享--解决你遇到90%的Git问题
  • 一.“版本控制系统”问世
    • 1.什么是"版本控制"？确保不同人所编辑的同一东西能集体得到更新，并在必要时候可以对没一个修改状态进行：查看、恢复
    • 2.为什么会有"版本控制系统"？==>"工具"的产生都是为了解决"生产"上的烦恼==>
      • 各自复制+重命名版本号：名字混乱(起名字真的很难起)、被覆盖了就"完蛋"了、不能协作
      • 本地版本系统:RCS【类似"系统补丁"，在硬盘上保存"差集"，回退时进行大量计算比较得出不同】--不能协作，个人电脑容易"犯傻"，遇到了问题：如何让不同人在不同系统上的开发者协同工作？
      • 集中式版本管理【有一个单一的集中管理的服务器，保存所有文件的修订版本，而协同工作的人们都通过客户端连到这台服务器，取出最新的文件或者提交更新】：CVS、SVN
      • 分布式版本管理【客户端并不只提取最新版本的文件快照，而是把代码仓库完整地镜像下来。 这么一来，任何一处协同工作用的服务器发生故障，事后都可以用任何一个镜像出来的本地仓库恢复，更进一步，许多这类系统都可以指定和若干不同的远端代码仓库进行交互】：Git、Bazaar
    • 3.为什么会有"Git"?==>2005年，BitKeeper的商业公司与Linux内核开源社区的合作关系结束，Linux之父一气之下开发了自己的版本控制系统：速度、安全、简单的设计、对非线性开发模式的强力支持(允许成千上万的分支并行开发)、完全分布式、有能力高效的管理类似Linux内核一样的超大规模项目(速度与数据量)
  • 二.Git与SVN的比较
    • 一).优缺点：
      • SVN优缺点：
        • 优点：1.相比老式版本控制，解决了协同合作问题，让每个人在一定的程度上知道其他人的工作；2.权限管理方便，能做到非常"精细"的分配；
        • 缺点：
          • 1.中央服务器单点故障，如果中央服务器磁盘发生故障，且没做过备份或者备份得不及时的话，还会有丢失数据的风险,最坏的情况是彻底丢失整个项目的所有历史记录【不能保证所有的数据都已经有人提取出来】
          • 2.提交必须有网络连接--非本地版本库；
          • 3.权限不支持分支继承；
          • 4.迁移量大【仍然是记录版本间的"差异"】；
          • 5.SVN原理上只关心文件内容的具体差异--进行还原的时候需要反复计算差异得到结果
          • 6.SVN的工作区和版本库物理上分开，只能通过http、https、svn等协议访问"某一历史状态下的快照"
          • 7.服务器压力大，数据库容量暴增
      • Git优缺点：
        • 优点：
          • 1.只要你愿意，你的"仓库"就是所有人的仓库；
          • 2.从技术上讲：Git可能永远做不到类似SVN的路径授权(精细授权)--公司对代码库进行合理拆分，按库授权
          • 3.每一次提交都是对代码仓库的完整备份--快照模式==>所以Git在切换操作上能如此的"速度"
          • 4.工作区与本地仓库如影随行，不受"外界"干扰--你做你的，我做我的，必要的时候再提交到远程仓库合并更新；
          • 5.只要有一个人的代码是最新的，那么就不会丢失掉"所有提交记录
          • 6.Git使用SHA-1散列哈希值作为全球唯一的版本号，这样的好处：①对于一个分布式版本控制系统，每个人每次提交后形成的版本号都不会出现重复；②保证了数据的完整性【哈希值是根据文件内容+目录结构计算出来的，且这一数据验证是被放在Git的底层，所以想要瞒过Git进行操作几乎是不可能】
        • 缺点：代码保密性差，开发者可以克隆所有代码和版本信息
    • 二).区别：
      • SVN的工作区中"每一个目录"都包含一个名为.svn的控制隐藏目录，该目录的作用：①标识"当前工作区"和"版本库检出时"的状态关系；②包含了一份"检出时"的原始拷贝，当比较文件改动差异&&本地改动回退时，可直接参考原始拷贝进行操作不需要联网；③在目录下按文件搜索时，.svn下的文件原始拷贝会导致多出一倍的搜索时间和搜索结果；④.svn很容易在集成时引入产品中，从而导致Web服务器安全隐患
      • Git只在工作区的根目录下有"唯一"的.git的子目录，这个.git目录就是"完整的版本库"本身,存放了所有的元数据与对象数据【一旦你删除，就是破坏了整个版本库】，除了和其他人交换数据外，任何版本库相关的操作都在本地完成【本地操作，避免了冗长的网络延迟--速度快】；版本库"可以"脱离工作区而存在，成为"赤裸"版本库--搭建git仓库(git init --bare)，但是工作区"不可以"脱离版本库而存在
        • 思考：一旦你.git，就是破坏了整个版本库，那么如何保证Git数据安全呢？方法①【本地的数据镜像化】：在A磁盘分区中创建"赤裸"版本库( --bare) ，然后在另B磁盘分区中克隆A中的版本库作为工作区，把B工作区的提交定时的PUSH到A分区的版本库；方法②：其他人定时clone项目并pull备份
      • SVN的全局版本好相较于CVS的每个文件都独立维护一套版本号相比，是一个非常大的进步，看似简单的进步实际标明：SVN提供支持"事务提交"--每一个"事务处理"都具备整个版本库全局唯一的版本号，但SVN的版本号是"连续"的，可以预判到下一个版本号；Git的版本号就更"碉堡"了，是全球唯一的(SHA-1碰撞+数据长度一致的情况下才会有问题)，Git每一次提交，会对"文件内容和目录结构"计算出一个SHA-1散列哈希值【40位的十六进制字符串】作为版本号，这也增加了预判下一个版本号的难度(几乎是不可能);--【Git 一般使用从左面开始的7位字串作为简化版本号，只要该简化的版本号不产生歧义（只要不会出现重复的，你也可以使用更短的版本号）即可】【关于哈希碰撞问题，Git的解决方法：How would Git handle a SHA-1 collision on a blob?：https://stackoverflow.com/questions/9392365/how-would-git-handle-a-sha-1-collision-on-a-blob/9392525#9392525】
      • SVN可以将整个库检出到工作区，也可以将某个目录检出到工作区(每个文件夹里都有一个.svn)，对于一个庞大、臃肿的版本库用户来说，部分检出是非常方便的，但是Git只能全部检出，不支持按照目录进行部分检出【只有在根目录下有一个.git文件，但是Git可以通过子模块处理版本库模块化的问题从而解决这个问题】
      • SVN的分支是通过svn cp命令，即：带历史的拷贝(但这是一种廉价拷贝：并不是拷贝所有数据，而是建立一个已存在目录树的入口，这也导致了SVN的分支之间没有完全"隔离")；Git中的分支实际上仅仅是一个包含所指对象校验和的文件，因此创建和销毁一个分支都非常的廉价(Git新建一个分支其实就是向一个文件写入41个字节：40位版本号+1个换行符)！Git的分支是完全隔离的；
      • SVN不会记录任何合并操作，当你提交本地修改，版本库并不能判断改提交来源于svn merge还是你手工修改之后svn commit的；而Git则可以很清晰的分辨出你的提交来源是：git commit 还是 git merge 亦或是git push
      • SVN一旦完成向服务器的数据提交，就没法从客户端追回，只能在后续提交中覆盖修正【集中式】;Git是分布式的，代码库允许任意的修改和丢弃【git rebase 、 git filter-branch、git reset】，因此Git可以修改和重构历史提交，且非常灵活，但是SVN修改历史提交只能由管理员来完成(在服务端进行撤销和修改，代价大)，但SVN能修改任意节点的提交说明，而Git只能修改最后一个commit的信息，且修改后将产生一个新的提交(commit号会变)；
    • 推荐阅读《SVN与Git的比较》系列：http://blog.sina.com.cn/s/blog_765348af0102uyo9.html
  • 三.Git 协议
    • 一)、本地协议【Local Protocol】==>【cd xxx】git init --bare test.git ==> git clone /xxx/test.git
      • 1.优点：1).简单；2).直接使用现有文件权限和网络访问权限；
      • 2.缺点：1).配置共享文件系统难；2).不方便多位置访问;3)."远程磁盘"方式访问，配置不便，访问慢；4).文件协议不保护仓库意外的损坏
    • 二)、"哑"HTTP协议
      • 1.Git 1.6.6版本之前只有一个方式可用，比较简单且通常是"只读模式"，而且是谁都可以读～
    • 三)、"智能"HTTP协议
      • 优点：
        • 1.Git 1.6.6版本引入；
        • 2.运行在标准的HTTP/S端口上并且可以使用各种HTTP验证机制
        • 3.即支持 "git://" 协议一样设置匿名服务；也可以像SSH协议一样提供传输时授权和加密数据
        • 4.目前使用最流行的：可用性强
          • 不同访问方式，只需要一个URL以及服务器只在需要授权时提示输入授权信息
          • 相比SSH协议，设置简单，免去设置公钥
          • HTTP协议传输也非常的快和高效
          • 可以在HTTPS协议上提供只读版本库的服务；甚至可以让客户端使用制定的SSL证书
          • HTTP/S协议被广泛使用，一般的企业防火墙都会允许这些端口的数据通过
      • 缺点：
        • 架设HTTP/S协议的服务端比SSH协议难；
        • 如果你在HTTP上使用需授权的推送，管理凭证会比使用SSH秘钥认证麻烦
    • 四）、SSH协议
      • SSH协议是一个需要验证授权【采用公私密钥】的、普遍、建设简单、使用简单的协议；
      • git clone ssh://user@server/project.git 或[scp方式] git clone user@server:project.git
      • 优点：
        • 1.建设SSH协议访问服务简单，而且大多数环境下已经支持SSH访问
        • 2.SSH访问安全--所有传输数据都要经过授权和加密
        • 3.SSH协议很高效，传输前会尽量的亚索数据【其他的协议也会】
      • 缺点：
        • 1.不能匿名访问【不利于项目开源】
        • 如果你要同事提供匿名只读访问和SSH协议，那么需要搭建SSH服务和架设一个可以让他人访问的服务
      • 生成SSH公钥：ssh-keygen -t 加密方式[rsa|dsa] -C "youremai@email.com"：https://help.github.com/articles/connecting-to-github-with-ssh/
    • 五）、git协议
      • 包含在Git里的一个特殊的守护进程，监听9418端口，但是访问无需任何授权，没有任何安全措施【要么所有人都可以访问，要么谁也不能访问，而且只要有人知道项目URL都可以向项目推送数据】
      • 优点：Git协议是Git使用的网络传输协议中最快的
      • 缺点：缺乏授权机制；架设繁杂困难；监听端口为非标准端口，一般企业防火前不会开放；
    • 推荐阅读：https://git-scm.com/book/zh/v2/Git-内部原理-传输协议
  • 四.3张图了解Git工作原理
    • 一).Git 是一套内容寻址 (content-addressable) 文件系统
      • 工作区：.git文件夹所在的目录下的所有范围都是工作区(除了.git文件夹外)
      • 暂存区：执行git add <file>后，对象暂时存放的区域，是工作区和本地仓库的桥梁
      • 本地版本库：.git文件夹==>Git仓库的核心所在，存放了所有数据版本的变更
      • 远程版本库：与当前仓库有绑定管理的xxx.git文件目录 #一般会由一个中心服务器来担当
      • 推荐阅读《Git对象模型》：http://gitbook.liuhui998.com/1_2.html
    • 二).Git引用：的分支(本地、远程)、标签、HEAD、暂存区(index -- add)、储藏(stash)、对象(tree/blob/commit/tag)的存储......这些都是引用，都是一串SHA1字符串
      • *新版本的 Git 不再使用 branches 目录，description 文件仅供 GitWeb 程序使用*
      • 推荐阅读：https://git-scm.com/book/zh/v2/Git-内部原理-Git-引用

  • 五.Git 常用操作
    • 1.仓库
      • 克隆远程仓库：git clone [-o origin] [-b master] [--depth 1] git@gitlab.com:winz_mmc/mmc_php.git [dir] ==> 当仓库过大时，可以指定克隆分支与深度
      • 创建裸仓库：git init --bare xxx.git ==> 自我创建仓库进行学习
      • 远程仓库：
        • 查看远程仓库信息：git remote -v 查看当前仓库所绑定的远程信息详细信息【fetch | pull】
        • 获取远程仓库URL:git remote get-url <remote>
        • 添加远程仓库：git remote add <remote_name> <remote url>
        • 重命名远程仓库：git remote rename <old> <new>
        • 修改远程仓库URL:git remote set-url <remote_name> <new_url>
        • 删除远程仓库： git remote remove <remote_name>
      • 本地仓库：
        • 拉取本地仓库所有分支的最新远程信息：git pull --all
        • 推送当前分支到远程并设置默认追踪：git push -u <remote> <local_branch>:<remote_branch>
        • 查看当前分支与远程分支相差的log详细信息：git log <remote/branch>..<branch>
        • 清除本地无效的远程分支：git fetch -p --all
      • 标签引用
        • 在指定版本号上打上标签：git tag <tag_name> <commit>
        • 将标签推送给远程仓库：git push --tag
    • 2.分支
      • 创建并追踪分支：git branch -t <branch_name> <remote/branch>
      • 创建并检出：git checkout -b <branch_name> [-t(设置追踪)] [HEAD |commitID]
      • 切换分支：git checkout <branch_name>
      • 重命名分支：git branch -m <old_branch_name> <new_branch_name>
      • 删除本地分支：git branch -D <branch_name>
      • 删除远程分支：git push <remote> :<remote_branch>
      • 查看所有分支的最后一个提交：git branch -a -v
      • 合并：http://yanhaijing.com/git/2017/07/14/four-method-for-git-merge/
      • 查看所有分支git branch -a -v
      • 获取所有分支(本地+远程)：git branch -a
      • 删除除当前分支外的所有本地分支：git branch | grep -v '*' | while read branch; do git branch -D $branch; done
      • 删除所有已经merge到master的分支：git branch --merged master | grep -v '^\*\| master' | xargs -n 1 git branch -d
      • 设置分支追踪：
        • git branch --set-upstream-to=<origin>/<branch> <local_branch>
        • git branch -u <remote/branch> [HEAD|<branche_name>]
    • 3.工作区
      • 只暂存已经追踪了文件：git add -u
      • 交互式分块暂存： git add -p <file>【https://git-scm.com/docs/git-add】
      • 删除工作区与版本库中的文件：git rm <file>【删除文件及索引 == rm file +git add file】
      • 强制删除：git rm -f <file>【当文件有未提交的缓存区修改时，直接使用git rm 文件会报错】
      • 删除文件在版本库的记录但留存在工作区(该文件将不再被跟踪)：git rm --cached <file>
      • 重命名文件：git mv <old_file> <new_file>
      • 查看当前工作区和暂存区状态：git status
    • 4.暂存区
      • 查阅
        • 查看当前暂存区commit信息:git ls-files --stage
        • 显示冲突的文件：git ls-files -u [git status -s]
        • 显示标记为冲突已解决的文件：git ls-files -s
      • 提交
        • 提交暂存区信息：git commit -m "message"
        • 修改最后一次提交变更：git commit --amend -m "message"
      • 储藏
        • 储藏工作区与暂存区文件：git stash save -u "message"
        • 储藏当前文件，并保留暂存区文件：git stash save --keep-index
        • 交互式部分储藏：git stash -p
        • 删除储藏栈指定记录：git stash drop stash@{num}
        • 查看当前储藏栈列表：git stash list
        • 应用指定储藏：git stash apply stash@{num} 【git stash pop -- 默认stash@{0}】
        • 基于指定储藏记录创建分支：git stash branch <branch_name>
        • 写完代码，发现不在对应分支：①git stash save -u "message";②git checkout <branch_name>;③git stash pop【*stash不要用太多，只是用于临时存储*】
    • 5.合并
      • "变更集"合并 -- merge
        • 合并其他分支代码：git merge <other_branch>【允许快进式合并，如果有冲突则产生新提交】
        • 强制非快进式合并：git merge dev --no--ff
        • 使用squash方式合并，把多次分支commit历史压缩为一次：git merge <branch> --squash
          • 注意：该操作合并之后需要额外commit一次
          • 坑：这种方式合并不会讲删除操作合并进来【在"黄色分支"执行git rm file1然后commit,在蓝色分支使用square方式合并，并不会删除file1】
        • 当有冲突时：
          • 停止当前merge操作：git merge --abort
          • 继续当前merge操作：git merge --continue
      • "变基"合并 -- rebase
        • 将当前分支【feature】以补丁的方式在指定分支上重演合并一次：git rebase <other_branch>
          • 继续当前变基操作【已经解决了当前冲突】：git-rebase --continue
          • 停止当前变基操作【停止后，"和平分手"】：git-rebase --abort
          • 丢弃本地冲突的commit继续当前变基操作【高危操作】：git-rebase --skip
          • 修改当前编辑操作： git-rebase --edit-todo
          • rebase黄金法则：绝不要在公共的分支上使用它！！！==>rebase操作会将当前分支的改动提交信息全部置顶提交！！
            • 比如说，如果你在develop分支上，rebase到你的feature分支上会发生什么？
              • 1. rebase会将当前分支(Develop分支)上的所有的改动提交都"重新演进一次",并将其作为一个新的"补丁方式"的提交移到了feature分支后==>但该改变"仅仅"还在你本地发生的，其他人还在Develop分支上做着改动！
              • 2.因为rebase引起了新的提交，Git会认为你的develop分支和其他人的develop已经分叉了，而要同步两个develop分支的唯一办法是把它们merge到一起，这就导致了："一个额外的合并提交"和"两堆包含同样更改的提交"，让人感到莫名其妙！
        • 自由选择合并点--"摘樱桃模式"：git cherry-pick commitID

    • 6.如何在 Git 里撤销(几乎)任何操作撤销
      • Git撤销示意图
      • 1.代码已经push到中心仓库"公共分支"上：git revert <commitID>【严禁采用reset代码再push -f，会覆盖掉别人代码的~】
      • 2.更改"最近一次"提交且代码"没有推到公共分支"：git commit --amend [-m <message>]
      • 3.代码还在工作区，没进行git add 操作：git checkout HEAD <file>
      • 4.代码已经进行add操作，但是没进行commit操作
        • 撤销暂存区修改并保存到工作区：git reset HEAD <file>
        • 撤销丢弃"所有文件"修改： git reset HEAD --hard
      • 5.已经进行commit操作，但是还没有push
        • 撤销丢弃已经commit的操作：git reset HEAD^ --hard
        • 撤销commit操作并留存在暂存区：git reset HEAD^ --soft
      • 6.撤销N久的一个操作：
        • ①git log --pretty=oneline，找到对应的commitID，然后git revert commitID
        • ②git reflog【能找到git log没有的信息，但有时限】，找到对应commitID再revert
      • 推荐阅读：http://blog.jobbole.com/87700/
    • 7.git中所有(几乎)忽略文件的方式：
      • 已跟踪文件：
        • 使用.gitignore文件忽略未跟踪文件【整个项目组都要同步的忽略】
        • 全局忽略：git config --global core.excludesfile path/global_gitignore_file【例如vender目录】
        • 个人暂时忽略未跟踪文件，不需要同步到中心仓库：vi .git/info/exclude文件 
      • 已经跟踪的文件：
        • 忽略：git update-index --assume-unchanged FILE
        • 解除忽略：git update-index --no-assume-unchanged FILE
        • update-index使用场景：①针对对象是已跟踪文件；②暂时忽略跟踪；③在为解除忽略并同步过信息前强制推送会导致严重的覆盖后果
    • 8.差异比较【只会比较跟踪(git add)了的文件】
      • 比较当前工作区和暂存区的不同：git diff
      • 比较指定文件在工作区和暂存区的不同：git diff <file>
      • 比较暂存区和仓库的不同：git diff --cached
      • 比较指定文件在暂存区和仓库的不同：git diff --cached <file>
      • 比较任意两个引用：git diff <commit1> <commit2>【分支|tag|commit都是特定SHA1的"引用"】
      • 比较版本变化时, 只高亮行内改动, 而非整行：git diff --word-diff
      • 只显示有差异的文件，不显示具体信息：git diff --stat
      • 忽略空白变更比较：git diff -w

  • 六.工作流程规范建议
    • 工作流程介绍
      • 集中式工作流【大部分公司应该都是这种方式】
      • 集成管理者工作流【开源项目的协作方式，开发者自行fork仓库，然后请求管理者拉取更新】
      • 司令官与副官工作流【就是在集成管理者工作流之上又加了一层管理者，适用庞大的项目=>Linux】
    • 分支策略：
      • 分支名规则【创建分支时，底层会执行git check-ref format进行强制检查】：
        • 分支名不能以减号"-"开头【"-"是git命令选项参数的选择标识】
        • 使用斜杠"/"分层的分支名不能以点"."开头
        • 分支名中不能包含连续的两个点"." [连续的两个点是有特殊意义的，代表分支区间]
        • 分支名中不能包含任何空格与其他空白字符、ASCII码控制字符、Git中有特殊意义的字符：~ ^ : ? * [
      • ERC分支规范对比与建议
        • ERC：master【稳定分支】，dev/develop【开发分支--相当于废了】，feature分支【修bug、日期发布、】、个人开发分支【全拼...】==>乱、分支多所以合并后的"提交树"非常扭曲，你想一下如果你看到下面这样的分支提交信息，作何感想？如果出了问题需要revert，你要怎么找到你的commit信息？
        • 建议：
          • 使用斜杠"/"对分支进行分类命名，不要全部都用feature-xxx
          • master -- 稳定发布分支【理论上master的代码是可以保证随时发布到线上】
          • dev/develop -- 开发分支（目前处于废弃状态）【可进行提交历史整合，用于保证master的提交清晰、明了】==>前后端应该是一致的或者是统一的，保证新建日期发布分支时的依赖是正确的
          • dev-pre-release 预发布分支（目前没有，预发布环境直接用master）；==> 起源于master，回归到master
          • 个人分支:personal/xxx（姓名全拼）；==>起源于dev，回归到dev或"日期发布分支"
          • 补丁分支：hotfix/xxx（修复功能点）；==>起源于master，回归到"预发布"或master
          • Bug修复分支：bugfix(或者issue)/xxx（修复功能点）;==>起源于dev，回归到dev或日期发布分支
          • 功能/需求分支：feature/xxxx（功能名）
          • 日期发布分支：release/xxx（当天发版日期）;==>起源于master，回归到预发布或dev分支
          • 用完的分支一定要及时删除：(1).分支太多比较乱；(2).代码过于陈旧，提交的时候目前99%会有冲突的【因为现在我们的迭代速度太快了】
        • 预发布不应该"等"所有人代码都没问题再从日期分支直接合并到预发布！！！可以让测试通知已经可以预发布的开发人员合并代码到预发布~
        • 为什么要倡导"规范"？当协作的人员越来越多的时候，能更加有效的避免代码冲突、避免提交树的杂乱无章、减小回滚成本
    • 为什么每个人都应该写好自己的commit信息？
      • 1. 提供更多的历史信息，方便快速浏览
      • 2. 可以过滤某些commit（比如文档改动），便于快速查找信息【commit信息即项目的change log】
      • 3. 可读性好，清晰，不必深入看代码即可了解当前commit的作用
      • 4. 为 Code Reviewing做准备
      • 5. 方便跟踪工程历史
      • 6. 方便进行git revert commitID操作
      • 7. 方便通过git来进行debug时的操作
      • 8. 提高整体项目质量与人工素质
